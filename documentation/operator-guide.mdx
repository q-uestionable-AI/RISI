## One interface for humans and agents

Every operational command supports `--format json`. The envelope always includes a schema version,
command, status, optional run identifier, structured data, and structured errors. Human-readable
output calls the same application service and safety kernel.

## Contracts

A run manifest declares an exact scenario path and digest, seed, adapter, decision provider,
policy, capability request, and resource limits. It cannot define grants or increase the built-in
profile ceilings.

Approval evidence contains the exact manifest SHA-256, reviewer identity, and approved capability
scope. It provides provenance and change detection, not authentication. Protect or sign it outside
an agent's writable context when a stronger boundary is required.

The isolated-target lifecycle uses three separate contracts. The target manifest binds transport,
certificate, credential fingerprints, API-path, and service identities. The campaign manifest
binds that target digest, the private-input inventory digest, geometry, retrieval settings, evidence
path, and expiry. Campaign approval binds both manifests, the inventory, exact capabilities,
purpose, expiry, and attempt ceiling. A manifest requests authority; it never supplies a grant.

## Stable exit codes

| Code | Meaning |
| --- | --- |
| `0` | Success |
| `2` | Invalid input or contract |
| `3` | Blocked by safety policy |
| `4` | Evidence or replay integrity failure |
| `5` | Execution or artifact failure |
| `6` | Deterministic resource budget exhausted |
| `130` | Cooperative interruption before evidence finalization |

## Evidence bundle

Each completed run contains the approved manifest, approval evidence, separated target and
evaluator scenario views, canonical state and events, execution summary, and report. Pure-read
bundles contain one final state, retrieval, context, decision, and assessment. Controlled CRAF
bundles contain one shared initial state, per-arm final states and outputs, per-arm evaluator
assessments, and a comparison assessment.

Controlled RISI-C bundles contain one evaluator-only initial snapshot and comparison, four
evaluator-only final states and traces, four strict observer views, and retained synthetic decision
artifacts. Observer views contain only the observer's own query, response identifiers and contents,
and result count.

`inventory.json` records the path, size, and SHA-256 of every file. Retain the inventory SHA-256
reported by `risi run` outside the bundle when an independent integrity anchor is needed.

The execution summary and run result report approved and consumed values for episodes, retrieval
calls, logical steps, scenario input bytes, source-memory records, and complete bundle bytes.
Budget exhaustion is an engineering result with its own status and exit code; it does not produce a
completed experimental bundle or a RISI/CRAF outcome.

`risi inspect <bundle> --format json` verifies the complete inventory before returning a closed
summary of run, manifest, scenario, policy, result, resource use, digests, and inventoried paths.
`risi compare <bundle-a> <bundle-b> --format json` verifies both bundles and returns exact equality
or stable differing evidence paths and semantic anchors.

The `craf-reference` manifest requests exactly three episodes and six retrieval calls. Its result is
`controlled_mechanism_recovered` only when the required control, vulnerable, and protected pattern
passes every evaluator invariant. This is a local synthetic engineering result, not a finding about
an external system.

The `risi-c-reference` manifest requests exactly four episodes and twelve retrieval calls. Its
result is `controlled_mechanism_recovered` only when the vulnerable pair reaches advantage `0.5`,
the pure-read ablation remains at `0.0`, the canary is absent from observer evidence, the probe
budget is exact, the shared counter is the sole vulnerable-pair state mediator, and all four region
decisions are safe.

## Autonomous operation

An agent may inspect capabilities, validate a human-approved manifest, execute it, verify and replay
its evidence, compare verified bundles, and propose an interpretation. Re-invoking a run
returns the existing verified result with `reused: true`; an incomplete, changed, or tampered final
path fails closed without overwrite or deletion.

A1 is synchronous. The process result is authoritative, interruption is cooperative, and automatic
retry count is zero. There is no status service, active cancel command, wall-clock experimental
deadline, or asynchronous job surface. Capability discovery reports these dispositions and the
absence of approval, manifest, evaluator, evidence-edit, evidence-delete, and human-acceptance
authority. Human review remains responsible for scope, authorization, release, and research claims.
Model prose is never a substitute for the evidence bundle or deterministic checks.

## Isolated-target campaign commands

The unreleased source exposes these commands under `risi campaign`:

- `prepare` verifies all bound contracts and writes one immutable prepared checkpoint without
  contacting the target;
- `preflight` performs one approved target health check and retains a numbered, non-overwriting
  result;
- `status` reads the retained checkpoint;
- `cancel` atomically requests cooperative cancellation under the same campaign authority;
- `execute` requires a matching E3-purpose approval before reading credentials, private inputs, or
  creating an evidence root; and
- `inspect`, `verify`, `compare`, and `replay` are model-free and do not contact the target.

Credentials are passed by file path, never as a CLI value. The API key and health token must
independently match their SHA-256 fingerprints in the target manifest. The exact credential object
contains `target_id`, `api_key`, and a separate `health_token`; neither secret is renderable by the
adapter. Target and campaign manifests contain no secret value. A full E1
request presented with E2 validation approval returns `BLOCKED_BY_POLICY` before secret, input, or
evidence-path access.

The campaign lifecycle is synchronous. A cancel record prevents the next request; an already
dispatched request may complete and is retained, but is never repeated. A service error, malformed
response, TLS failure, certificate mismatch, deadline, indexing error, or unlisted route causes a
closed failure and a retained deviation.
